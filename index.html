Spring IOC与AOP学习小结

发表于 2018-05-26 |

  过几天就是自己上台表述Spring框架的知识点及其功能（原理不敢说懂了了），所以也就Spring单独的学习了一下，从中就把自己所学的与自己的理解跟大家分享一下。
  个人认为Spring框架是一个非常NB的开源框架，因为她大大减少了我们开发J2EE所需要的配置和环境搭建，而且解决了从前开发J2EE的高度耦合、后期维护难等问题，实现了低耦合、高内聚的设计思想。并且她不仅仅提供了这些。
Spring从其名我们可以看出是春天的意思，正如其名一样，Spirng对于开发J2EE的开发人员就是带来了春天，而且越来越强大，其整合其他框架的融合性也是其一大亮点。
  Spring中的功能有很多，其中最重要的分别为IOC和AOP,功能甚是强大，让我流连忘返，但还是许多的东西不知其理，还得继续。
IOC:控制反转又称DI(依赖注入，她并不是一种技术实现，而是一种设计思想。在如何一个具有实际意义的项目中，我们会使用很多类来描述它们特有的功能，通过类与类之间相互协作来完成特点的业务逻辑，这个时候，每个类都需要负责管理与自己有交互的类的引用于依赖，代码将会变得异常的难以维护以及高度耦合，而IOC就是用来处理解决这个问题，IOC把这些相互依赖对象的创建、协调分配交给Spring容器去做处理，每个对象只需要关注自身的业务逻辑关系就行了。

依赖注入分为三种：
接口注入、setter方法注入、构造方法注入

以下几个问题非常精髓：
1.参与者都有谁？
对象、IOC/DI容器、某个对象的外部资源
2.依赖是什么？为什么需要依赖？
对象依赖IOC和DI容器，为什么对象依赖IOC/DI容器因为需要IOC/DI容器提供对象所需外部的资源
3.注入是什么？又注入了什么？
IOC/DI容器注入对象，注入了对象所需要的资源
4.控制反转：谁控制谁？控制什么？为什么叫反转？存在正转吗？
IOC/DI控制对象 、控制创建对象的实例，反转是相对于正转来说的，正转好比如说：当A类需要B类的东西时，你会使用什么办法来解决？那就是创建她的对象，也就是A类主动获取所需要的外部资源B类,这种情况就是正转，而什么是反向？那就是A类不主动获取外部资源B，而是被动等待，等待IOC/DI容器获取这个B类的实例，然后反向的注入到A类中
5控制反转与依赖注入是同一个概念吗？
控制反转与依赖注入是对一件事的不同描述，也就是描述的角度不同
控制反转：从应用程序的角度，应用程序依赖容器创建并注入所需要的外部资源
依赖注入：从容器的角度，容器控制应用程序，由容器的反向注入应用程序所需要的外部资源

上述几个问题对于IOC的大概做了一个提问，而其答案也是我们所需的，能让我们更好的、快速的了解IOC与DI,接下来让我们对Spring的另一个核心AOP做一个了解

AOP:全名Aspect-Oriented Programming，中文直译为面向切面(方面)编程，当前已经成为一种比较成熟的编程思想，可以用来很好的解决应用系统中分布于各个模块的交叉关注点问题。在轻量级的J2EE中应用开发中，使用AOP来灵活处理一些具有横切性质的系统级服务，如事务处理、安全检查、缓存、对象池管理等，已经成为一种非常适用的解决方案。 AOP中比较重要的概念有：Aspect、JoinPoint、PonitCut、Advice、Introduction、Weave、Target Object、Proxy Object等

对于AOP的，我在实际项目中所主要运用的主要是事务管理那一块，所以AOP其他的能够自己表达的真的很少，相信下次自己带来的是更加完善的关于AOP的学习心得，下次依旧！

自定义filter：
–AuthorizationFilter抽象类事项了javax.servlet.Filter接口，它是个过滤器。
public class Cbuer extends AuthorizationFilter {

    protected boolean isAccessAllowed(ServletRequest req,
            ServletResponse resp, Object mappedValue) throws Exception {

        Subject subject = getSubject(req, resp);  
        String[] rolesArray = (String[]) mappedValue;  

        if (rolesArray == null || rolesArray.length == 0) { //没有角色限制，有权限访问  
            return true;  
        }  
        for (int i = 0; i < rolesArray.length; i++) {  
            if (subject.hasRole(rolesArray[i])) { //若当前用户是rolesArray中的任何一个，则有权限访问  
                return true;  
            }  
        }  

        return false;
    }

}

学习java堆和栈

发表于 2018-04-03 |

今天带来的是删了又删，看了又看，还是没有总结出自己的话语，原因有多种，最重要的一点的就是自己并没有去认真的去对待这件事，为什么这样了，重要的事情说三遍：还是太年轻、还是太年轻、还是太年轻。

java的中的内存分为两种：“栈”和“堆”，栈通常是自己定义基本数据和对象的引用时保存的地方，而堆就是自己通过new一个新的对象或者数组保存的地方了，两者的差距也很大，据我初步了解，栈的速度远远快于堆，也就是说堆的速度不是很快，相对于程序而言，应该算慢了（中立），但java是一个面对对象的静态语言和强类型语言，用到堆内存的时候太多了，所以这个应该算java一个很大的缺点了。

再往详细说，“栈”内存定义一个变量时，栈就会把他放在它的内存中，当超过这个变量的作用域时，java会自动释放掉“栈”给这变量分配的内岑空间，并且这个内存空间可以立即作为别的内存使用。“堆”中分配的内存，由java虚拟机的自动垃圾回收期来管理。

hexo + GitHub搭建博客续集

发表于 2018-03-20 |

接着上一篇博客所说的，我们现在只能看到一个非常简陋的由hexo所默认的主题背景，所以我们现在就如何了解和更换博客主题做一个探讨。

首先我们得了解的是hexo中两个重要的配置文件（_config.yml），名字是一样的，只是放的位置不同,一个叫做站点配置文件，放在根目录文件夹下，另一个叫做主题配置文件，放在根目录themes文件下的每一个主题文件夹里面，这里我们说一下什么是主题文件夹，首先，当你安装hexo之后，默认你是有一个主题的，这个主题就是这个文件夹里面的所有的东西，一个文件夹代表一个主题，所以你现在应该明白了吧。

想要其他的主题就是通过下载获得然后去站点配置文件去更改，先不说如何去改主题，hexo根目录下的每一个文件到底是做什么的，我们现在去理解一下

以下引入Hexo的官方文档中的一些数据
.
├── _config.yml
├── package.json
├── scaffolds
├── source
| └── _posts
└── themes

_config.yml:网站的配置信息，你可以在此配置大部分的参数，更多的参数请自行了解。

package.json:应用程序的信息，好像是默认安装的，反正我没有删除，提示可以移除。

soaffolds:模版文件夹,当您新建文章时，Hexo会根据scaffold来建立文件。Hexo的模板是指在新建的markdown文件中默认填充的内容。
例如，如果您修改scaffold/post.md中的
Front-matter内容，那么每次新建一篇文章时都会包含这个修改。

hexo + GitHub 搭建一个博客

发表于 2018-03-19 | 
首先需要电脑下载两个必须软件Node.js(Node.js)

与git(Git - Downloads)，从网上随便找一下就有安装教程
查看是否安装成功在命令行输入以下：
node -v //查看Node.js 显示Node.js版本
git -v //查看git 显示我也看不明白的

假如上述的要求已经可以之后（必须是在完成之后），随便你去电脑上的如何一个页面

点击右键打开git bash(如何上述的git你安装完成了，应该是可以显示的)
在git bash页面输入以下命令行：(可以参考hexo官方的文档)
（https://hexo.io/zh-cn/docs/）
npm install -g hexo-cli //安装hexo
你一定要等到完成为止，就是跳回你的当前所在位置一串彩色字体，还有就是需要耐心等待，有时候会因为一些原因（我也不晓得）卡住，这时候看看天空也不错
hexo安装完成之后，你得在系统盘创建一个保存hexo的文件夹(名字随自己创建)，然后通过git bash打开，假如我把它放在我F盘的Boke文件夹下的hexo文件里面那么我通过
其一： cd /f/Boke/hexo

其二： 就是在hexo文件夹点击右键git bash也可以

然后就是把hexo的东西都放到这就文件里面,在上述打开的git bash里面输入
npm init

同样你还是要等待他自动跳回彩色字体也就是跳回当前所在位置，
安装完成之后你就可以看到你的hexo文件夹里面包括以下
（引用知乎上石先生的）
+ themes 存放主题的文件夹
+ source 博客资源文件夹
+ source/_drafts 草稿文件夹
+ source/_posts 文章文件夹
+ themes/landscape 默认皮肤文件夹

其实上述的东西在官方文档中也很详细

上述文件都有的话，代表你成功了一小半，接下来输入以下：
git g  //好像就是Tomcat添加Web项目但并没有发布
git s  //这个就是发布了，输入这个之后，你只有通过 Ctrl + c 来关闭，
所有在git bash里面不要用Ctrl + c，
可以通过邮件复制，粘贴
接下来就通过在本地浏览器查看自己的初步博客，输入：127.0.0.1:4000

有时候端口号会冲突，我没有碰到这个事情，所有提个醒（http://blog.liuxianan.com/windows-port-bind.html）
你可以自己进去看看
博客主题是hexo默认的，当然你也可以修改的：休息一下

到底用GitHub来搭建免费博客好不好？

发表于 2018-03-18 |

从开始了解如何去创建一个基础的个人博客网址的时候，自己觉得挺难的，
因为自己虽然学了Java一年多了，但是自认为学的不是很扎实，最近一直在学Web开发的框架知识，
但是学的比以前开心和认真，美中不足的是没有发布到服务器上面去，只能在本地发布玩玩，可怕。
从网上看到许多的如何搭建个人博客，自己虽然卡了几下子，但是还是弄了出来，
主要是用hexo + GitHub（粗制滥造都不为过）。

搭建GitHub后，但是自己又从知乎上看到几篇文章对我影响很大：

其一：用hexo + GitHub 搭建一个博客上真的太简单
真的在搭建完之后我感觉上挺简单的，虽然其中的内容并不是很多，
但是自己不就是想弄一个简单的东西吗（心态稳的一匹）？而且网上关于创建博客的文章多如牛毛，
想想太可怕，导致了我都想放弃了（岂能放弃，走自己的路，何必在乎他人的观点），
最重要我去年买的域名终于有了它用武之处，不然太浪费，你呢？

其二：用GitHub来搭建个人博客是否对GitHub带来弊端
GitHub是一个面向开源及私有项目的托管平台，只支持git,所以有了GitHub之称，其中不乏很多当前所最新的技术，所以是获取和学习知识以及快速的好地方，用来搭建个人博客感觉就是有点大材小用，想想一句古话：“杀鸡岂用宰牛刀”，虽然有其理，但现实呢？现实是残酷的，所以维持本心，就是最好的，你觉得呢？

其三：小白搭建出一个博客，难道就这样了？
搭建出一个博客，需要前端、后端，以及发布到服务器，虽然hexo+GitHub提供了太多的前端的框架，
以及我也不了解的知识，所以自己也不能详细说出来，但我从中看出一个道理，
就是你对一个东西感兴趣的时候，并不是说你把这个东西弄完了，也就不管不问了，
不要说要你去把其原理弄清楚，但至少去明白如何用什么去弄出来，因为这是你自己下了功夫去弄的，要对得起自己！

话说到这，其实说的只是在这次搭建博客之后的所想，但其做并没有分享出来，所以下次我会把自己搭建博客的一步步写出来,下次依旧.
